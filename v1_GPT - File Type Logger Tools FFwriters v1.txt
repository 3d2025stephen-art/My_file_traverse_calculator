        You said:

Why no tool can read everything index?


        ChatGPT said:

They /can/, but not in the sense people usually imagine.

“Everything” works because it builds its own index on NTFS drives using
the Master File Table (MFT). That gives it near-instant knowledge of
filenames and paths. But that model has a few hard limits that explain
why other tools don’t just “read Everything’s index” directly.

First, Everything’s index is proprietary and internal. It’s optimized
for speed, not interoperability. It is not exposed as a standard
database format that other programs can reliably query. So other tools
don’t “read Everything”; they would have to re-implement the same
scanning/indexing logic themselves or use Windows APIs directly.

Second, file indexing is not a single universal layer in Windows.
Different tools rely on different sources:

  *

    NTFS MFT scanning (WizTree style)

  *

    Full filesystem traversal (WinDirStat style)

  *

    Windows Search index (a separate system index, slower, more
    metadata-rich)

  *

    Custom databases (Everything, third-party catalogs)

These are fundamentally different data models. One is a raw file ledger
(MFT), another is a search-optimized metadata index, another is a
recursive scan snapshot. There is no single “master index” that
everything plugs into.

Third, indexing is expensive in different ways depending on what you
want. Everything is extremely fast for names, but it does not deeply
analyze content, file types beyond extension, or historical changes
unless configured. A “universal index” that includes content parsing,
type classification, deduplication, and history would become huge, slow,
and constantly updating.

Fourth, external drives complicate everything. Your WD USB drives:

  *

    disconnect and reconnect

  *

    change drive letters

  *

    may not be always mounted

  *

    may sleep or power down

A persistent global index across that environment becomes fragile unless
tightly managed.

So the real answer is: there is no shared indexing layer because Windows
never standardized one, and because different tools intentionally
optimize for different trade-offs (speed, depth, persistence, portability).

In practice, people approximate a “global index” by combining:
Everything → fast file search
WizTree/TreeSize → disk structure + size analysis
Windows Search → content-aware search (limited but integrated)
